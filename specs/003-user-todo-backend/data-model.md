# Data Model: Персональні завдання

## Todo

Persistent task owned by exactly one authenticated user.

### Storage

- Database: `products_shop`, selected by `MONGODB_URI`.
- Collection: `todos`.
- Model name: `Todo`.

### Stored fields

- `_id`: MongoDB ObjectId, generated by the database.
- `title`: string, required, trimmed, length 1–200.
- `userId`: string, required, immutable owner identity derived from the authenticated session.
- `completed`: boolean, required with default `false`.
- `createdAt`: Date, generated by Mongoose timestamps.
- `updatedAt`: Date, generated and maintained by Mongoose timestamps.

### Indexes

- `{ userId: 1, createdAt: -1 }`: supports complete owner-scoped lists ordered newest first.

No uniqueness constraint exists for title; multiple users and one user may create tasks with identical titles.

### API representation

```text
{
  id: string,
  title: string,
  userId: string,
  createAt: ISO-8601 string,
  completed: boolean
}
```

Transform rules:

- `_id` becomes `id`.
- `createdAt` becomes the existing compatibility field `createAt`.
- `_id`, `__v`, `createdAt`, and `updatedAt` are not exposed.
- `userId` remains in the canonical response for compatibility, but never determines authorization on the client.

### Validation rules

- Create rejects non-string, empty, whitespace-only, and over-200-character titles.
- Title is trimmed before persistence.
- Create ignores client-provided `userId`, `completed`, `createAt`, `createdAt`, and `_id`.
- Patch permits only `title` and `completed`.
- Patch with `title` applies the same trim and length constraints as create.
- Patch rejects an empty update body or unsupported field-only body.
- `userId`, `_id`, and timestamps cannot be changed after creation.

### State transitions

```text
absent
  -> active(completed=false)   [create]

active(completed=false)
  -> active(completed=true)    [complete]
  -> active(new title)         [rename]
  -> absent                    [delete]

active(completed=true)
  -> active(completed=false)   [reopen]
  -> active(new title)         [rename]
  -> absent                    [delete]
```

Every transition requires the same actor ID as `userId`.

## User

Existing persistent account; its schema does not change.

### Relationship

- One User owns zero or more Todo documents.
- Each Todo stores one stable user identifier from the verified NextAuth session.
- Deleting users and cascading task cleanup are outside this feature.
- An orphan task is never returned because every query is scoped to the currently verified user ID.

## Authenticated Actor Context

Request-scoped identity, not persisted.

### Fields

- `userId`: non-empty string obtained by Next.js from the verified server session.
- `internalAuthorization`: secret bearer credential shared only by Next.js server and backend.

### Rules

- Missing or invalid internal authorization rejects the request.
- Missing or malformed user ID rejects the request.
- Browser-controlled query and body fields cannot override actor identity.
- After middleware verification, controllers read actor identity from request-local server context.

## Task List

Derived collection, not separately persisted.

### Query definition

- Filter: `userId` equals authenticated actor ID.
- Sort: `createdAt` descending.
- Empty result: `[]`.
- No pagination in this feature.

### Consistency rules

- A successful create returns the canonical persisted Todo used to replace the optimistic item.
- A successful patch returns the canonical updated Todo.
- A successful delete returns the removed canonical Todo or a stable deleted identifier according to the REST contract.
- A missing and a foreign Todo produce the same not-found outcome.
